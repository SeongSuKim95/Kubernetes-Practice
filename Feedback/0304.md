Docker / K8s, 두 기술 사이의 '연결 고리'와 '실제 체감 효과'
- docker compose, docker swarm, ...
- 왜 docker로 다 운영하면 되지, k8s가 필요한 것인가?

Docker Container와 Kubernetes Pod의 관계 (왜 Pod인가?)
- Docker의 기본 단위인 'Container'와 K8s의 기본 단위인 'Pod'의 개념적 차이
- K8s가 컨테이너를 직접 다루지 않고 Pod라는 래퍼(Wrapper)를 사용하여, 네트워크와 볼륨을 공유하는 이유
- Docker에 익숙한 수강생이 K8s를 처음 접할 때 가장 먼저 부딪히는 의문이 "왜 docker run처럼 컨테이너를 바로 띄우지 않는가?"입니다. Node.js 기반의 API 서버나 프론트엔드 앱 등 다수의 서비스가 맞물려 돌아가는 SaaS 플랫폼을 구성할 때, 밀접하게 연관된 컨테이너들을 하나의 논리적 호스트로 묶어주는 Pod의 개념을 짚어주어야 K8s의 기본 사상을 명확히 이해

쿠버네티스는 Docker를 직접적으로 이용(실행)하지 않습니다
- K8s는 왜 Docker를 버렸는가? (Dockershim의 퇴출)
- 과거(v1.23 이전)에는 K8s의 Worker Node에 있는 Kubelet이 컨테이너를 실행하기 위해 진짜로 Docker Daemon을 호출했습니다. 하지만 여기에는 문제가 있었습니다.
    - Docker는 너무 무겁습니다: Docker는 단순히 컨테이너를 '실행'하는 기능뿐만 아니라, 이미지를 빌드하고, 네트워크를 구성하고, 볼륨을 관리하는 등 개발자에게 편리한 수많은 기능을 뭉쳐놓은 '거대한 프로그램(Monolithic)'입니다.
    - K8s에게 필요한 건 오직 '실행'뿐: K8s 입장에서는 오케스트레이션을 본인이 다 알아서 할 테니, 빠르고 가볍게 컨테이너를 띄우고 지우는 순수 실행 엔진(Runtime)만 필요했습니다.
    - CRI와 containerd의 등장: 그래서 K8s는 CRI(Container Runtime Interface)라는 표준 규격을 만들었고, 무거운 Docker 전체를 거치는 대신 Docker의 핵심 실행 엔진인 containerd를 Kubelet이 직접 호출하는 방식으로 아키텍처를 최적화했습니다. (이로 인해 Docker와 통신하던 dockershim이라는 모듈이 K8s 1.24 버전부터 완전히 삭제되었습니다.)
- 그럼 우리가 배운 도커는 이제 쓸모없나요?"
    - "어? 그럼 제가 로컬에서 docker build로 만든 이미지는 K8s에서 못 돌리나요?"
    - 이미지는 국제 표준(OCI)입니다: Docker로 만든 컨테이너 이미지는 Docker만의 독자적인 포맷이 아니라 OCI(Open Container Initiative)라는 오픈소스 국제 표준 규격입니다.
    - 표준은 어디서든 돌아갑니다: K8s가 사용하는 containerd 역시 이 OCI 표준을 완벽하게 지원합니다.
    - 역할 분담: 즉, 개발자는 여전히 내 PC에서 익숙한 Docker CLI로 코드를 빌드하고 이미지를 만듭니다. 하지만 운영 서버(K8s 클러스터)에서는 무거운 Docker Daemon 대신, 더 가볍고 빠른 containerd가 그 이미지를 넘겨받아 군말 없이 실행해 주는 구조입니다.
- Container Runtime
    - 실제 컨테이너를 생성하고 실행하는 엔진입니다
    - 과거에는 Docker를 직접 사용했지만, 현재는 성능과 경량화를 위해 Docker의 핵심 실행 엔진인 containerd나 CRI-O 같은 표준 런타임을 주로 사용합니다. (물론 여러분이 만든 Docker 이미지는 100% 호환되어 똑같이 실행됩니다!)

선언형 배포의 실체: docker run vs YAML
- Docker CLI에서 명령어(docker run -d -p 8080:80 my-app)를 타이핑하여 실행하는 방식(명령형)과, K8s에서 인프라의 원하는 상태를 문서(YAML)로 작성해 kubectl apply -f로 제출하는 방식(선언형)의 실질적인 차이를 시각적으로 비교
- 2장에서 '선언적 API'라는 철학을 다루었지만, 코드로 직접 보기 전까지는 추상적으로 느껴질 수 있습니다. 인프라와 배포 설정 자체를 파일(YAML)로 만들어 Git으로 버전 관리(IaC, GitOps)할 수 있다는 점을 보여주면, 단순한 서버 관리 도구가 아니라 **"운영 환경 자체를 코드로 관리하는 플랫폼"**이라는 K8s의 진가를 확실히 각인시킬 수 있습니다.

장애 복구 시나리오: 수동(Docker) vs 자동(Kubernetes)
- 운영 중인 컨테이너가 갑자기 다운되었을 때, 순수 Docker 환경에서 관리자가 수동으로 대처해야 하는 과정과 K8s의 Control Loop가 이를 자동으로 감지하고 복구하는 과정
- 현재 1.2절에 오케스트레이션의 필요성이 잘 정리되어 있지만, 초심자에게는 다소 추상적으로 들릴 수 있습니다. "새벽 3시에 메인 서버 컨테이너가 죽었다면?" 같은 구체적이고 현실적인 장애 상황을 제시하면, 앞서 설명한 선언적 API(Desired State vs Current State)가 왜 그토록 강력한 무기인지 직관적으로 와닿게 됩니다

개발자 관점의 3대 핵심 리소스 (Pod, Deployment, Service)
- K8s 클러스터의 인프라적 구성요소(Control Plane, Worker Node 등) 외에, 개발자가 실제로 서비스를 배포하기 위해 가장 많이 다루게 될 3가지 논리적 객체의 역할(Pod: 실행, Deployment: 갯수 유지 및 배포 전략 관리, Service: 고정 IP 제공 및 로드밸런싱)을 설명(큰 그림)

API Server를 중심으로 한 이벤트 기반 통신 (직접 제어 vs 상태 구독)
- Scheduler와 Controller Manager의 명확한 역할 차이(Scheduler는 '배치'만, Controller는 '개수/상태 유지'만)를 정의
- 노드나 파드에 직접 명령을 내리는 것이 아니라, API Server(etcd)의 상태를 '구독(Watch)'하다가 변경 사항이 생기면 API Server에 업데이트를 요청하는 방식임을 설명
- 실제 파드의 실행과 상태 보고는 노드의 Kubelet이 API Server와 통신하여 이루어진다는 점

Control Plane의 고가용성(HA)과 SPoF 해결 (단일 서버인가, 다중 서버인가?)
- Control Plane이 단일 서버로 존재할 때 발생하는 SPoF(단일 장애점) 문제를 쿠버네티스가 어떻게 해결하는지 설명합니다. 실제 운영 환경에서는 3대 이상의 다중 마스터 노드(Multi-Master)로 구성되며, 앞단에 로드밸런서를 두어 여러 대에 떠 있는 API Server로 트래픽을 분산시킨다는 점을 짚어줍니다.
- "마스터가 죽으면 전체 서비스가 멈추는 것 아닌가?"는 아키텍처를 볼 때 가장 먼저 떠오르는 핵심적이고 당연한 의문입니다. Control Plane 자체가 무중단 운영을 위해 어떻게 이중화/삼중화되어 있는지 이해해야, 쿠버네티스가 왜 엔터프라이즈급 인프라 표준으로 자리 잡았는지 완벽하게 납득할 수 있습니다.

내 터미널과 클러스터의 연결 고리, kubectl의 동작 경로
- kubectl이 어떻게 그 수많은 마스터 노드 중 올바른 API Server를 찾아서 명령을 수행시키는지 설명합니다. 사용자 인증 정보와 API Server의 주소(로드밸런서 도메인 또는 IP)가 담긴 kubeconfig 파일의 역할을 소개하고, 모든 명령이 결국 REST API 형태로 API Server에 전달된다는 점을 요약합니다.
- 복잡한 내부 컴포넌트를 배우고 나면, "그래서 내가 엔터를 쳤을 때 정확히 어디로 요청이 날아가는가?"에 대한 물리적인 경로가 궁금해집니다. 이 부분을 짚어주면 이론이 실제 실무(CLI 조작)와 연결됩니다.
