# 03/11 질문 답변

## 1) Docker Swarm이 안쓰이는 이유?
Swarm이 “못 쓰는 기술”이라기보다, 업계 표준/생태계가 Kubernetes로 강하게 이동하면서 실무 선택이 Kubernetes로 쏠린 영향이 큽니다.

- **운영 기능의 깊이 차이**: Kubernetes는 리소스 요청/제한, 오토스케일링, 네트워크 정책, RBAC, 스토리지 추상화 등 운영에서 필요한 정책 도구가 풍부합니다.
- **생태계 중심 이동**: Helm/ArgoCD/Prometheus/Service Mesh/Operator 등 도구/레퍼런스가 Kubernetes를 중심으로 확장되며 인력풀도 함께 쏠렸습니다.
- **결과**: 규모가 커질수록 Kubernetes가 “표준 선택지”가 되는 경우가 많아졌습니다.

## 2) ManagerNode가 Container 단위의 health check를 하지 않는 이유?
Swarm에서 Manager의 역할은 “컨테이너 내부를 직접 검사하는 감시자”라기보다, **Desired State(예: replicas 수)를 유지하도록 스케줄링/조정하는 Control Plane**에 가깝습니다.

- **관찰 책임 분리**
  - 실제 컨테이너(Task) 실행/상태 관찰은 Worker 노드의 Docker Engine이 담당하고,
  - Manager는 보고된 상태를 바탕으로 “부족하면 다시 스케줄링” 같은 조정에 집중합니다.
- **이렇게 나누는 이유(장점)**
  - 확장성: 중앙이 모든 컨테이너를 세밀하게 점검하지 않아도 규모가 커져도 부담이 덜함
  - 장애 격리: 컨테이너 상태는 로컬(Worker)에서 관찰하는 편이 정확하고, 중앙의 오탐/미탐 리스크를 줄임
  - 정책 단순화: Swarm은 “서비스 복제 수 유지” 같은 목표에 집중하기 쉬움

## 3) Overlay network Layer에 대한 자세한 설명
Overlay 네트워크는 “서로 다른 호스트에 있는 컨테이너들이 같은 네트워크처럼 통신”하도록 만드는 계층입니다.

- **핵심 아이디어**: 컨테이너/서비스는 내부 IP로 통신하는 것처럼 보이지만, 실제 패킷은 호스트 간에 **캡슐화(예: VXLAN)** 되어 전달되고 목적지 호스트에서 풀려 컨테이너로 도착합니다.
- **효과**
  - 멀티 호스트 환경에서 서비스 간 통신을 단순화
  - 서비스 디스커버리/로드밸런싱과 결합해 운영 편의성 확보

## 4) Docker Swarm과 Kubernetes에서는 클러스터 단위의 이중화 기법있나?
**있습니다.** 둘 다 “관리 계층을 다중화하고, 상태 저장소는 쿼럼(과반) 기반 합의로 운영한다”는 큰 방향은 유사합니다.

- **Docker Swarm**
  - Manager 노드를 홀수 개로 두고(Raft), **과반이 살아있을 때** 클러스터 상태 변경/스케줄링이 정상 동작
- **Kubernetes**
  - Control Plane을 다중화하고(API Server 앞단 LB), 상태 저장소인 etcd를 홀수 개로 구성해 **쿼럼 기반**으로 운영
