# 03/04 질문 답변

## Q1. Docker / K8s, 두 기술 사이의 '연결 고리'와 '실제 체감 효과'는 어느 정도인가?
- docker compose, docker swarm 와 같은 docker만으로 운영해도 될 것 같은데, k8s가 왜 필요한 것인가?
- 예시와 함께 설명

### 답변
- **연결 고리**: Kubernetes도 결국 “컨테이너(이미지)를 실행”합니다. 로컬에서 `docker build`로 만든 이미지는 OCI 표준이라 레지스트리를 통해 Kubernetes에서도 그대로 실행됩니다.
- **K8s가 필요한 이유(체감 포인트)**: 규모가 커질수록 “사람이 판단/수동 조치하던 운영”을 **정책과 자동화**로 바꾸는 지점에서 차이가 커집니다.
  - 자동 확장(HPA), 리소스 요청/제한, 롤링 업데이트/롤백, 네트워크 정책, RBAC, 스토리지 추상화, GitOps 등

## Q2. Docker Container와 Kubernetes Pod의 관계 (왜 Pod인가?)
- Docker의 기본 단위인 'Container'와 K8s의 기본 단위인 'Pod'의 개념적 차이
- K8s가 컨테이너를 직접 다루지 않고 Pod라는 래퍼(Wrapper)를 사용하여, 네트워크와 볼륨을 공유하는 이유
- "왜 docker run처럼 컨테이너를 바로 띄우지 않는가?"

### 답변
- **Container**: Docker에서의 기본 실행 단위(각자 네트워크/파일시스템/생명주기).
- **Pod**: Kubernetes에서의 최소 스케줄링 단위(“함께 살아야 하는 컨테이너 묶음”).
- Pod를 쓰는 이유
  - 같은 Pod 내 컨테이너들이 **네트워크(예: localhost)** 와 **볼륨**을 공유
  - 사이드카(로그 수집기/프록시 등)처럼 “항상 같이 떠야 하는 컨테이너”를 한 덩어리로 배치/관리

## Q3. 쿠버네티스는 Docker를 직접적으로 이용(실행)하지 않는다. K8s는 왜 Docker를 버렸는가? (Dockershim의 퇴출)
- (v1.23 이전) Kubelet이 Docker Daemon을 호출했었음
- Docker는 무겁고, K8s에게 필요한 건 '실행' 뿐
- CRI와 containerd 등장, dockershim은 1.24부터 삭제

### 답변
- Kubernetes가 버린 건 “Docker 이미지”가 아니라, **Kubelet → Docker Daemon** 경유 구조(dockershim)입니다.
- Kubernetes는 **CRI(Container Runtime Interface)** 표준을 만들고, Docker 전체 대신 **containerd/CRI-O 같은 런타임**과 직접 통신하도록 바꿔 실행 경로를 단순/경량화했습니다.

## Q4. 그럼 우리가 배운 도커는 이제 쓸모없나요?
- "어? 그럼 제가 로컬에서 docker build로 만든 이미지는 K8s에서 못 돌리나요?"
- 이미지는 국제 표준(OCI)이고 containerd도 지원
- 역할 분담(개발: docker / 운영: containerd)
- Container Runtime이란?

### 답변
- **Docker는 여전히 유용**합니다(특히 개발 단계).
  - 개발자는 Docker로 이미지 빌드/로컬 실행/레지스트리 푸시를 합니다.
  - 운영(K8s)에서는 containerd 같은 **컨테이너 런타임**이 그 이미지를 받아 실행합니다.
- **OCI 표준**: Docker 이미지 포맷은 OCI 표준 기반이라 Kubernetes 런타임에서도 호환됩니다.
- **Container Runtime**: 이미지를 pull하고 컨테이너를 실제로 생성/실행/종료하는 실행 엔진입니다.

## Q5. 선언형 배포의 실체: docker run vs YAML
- Docker CLI에서 명령어(docker run ...)로 실행(명령형) vs YAML로 원하는 상태를 제출(선언형)
- YAML을 Git으로 버전관리(IaC, GitOps)

### 답변
- **명령형**: “어떻게 할지”를 실행(`docker run ...`). 실행 후 상태 유지는 사람이 챙기는 쪽에 가깝습니다.
- **선언형**: “무엇이 되길 원하는지(Desired State)”를 제출(YAML)하면, 컨트롤 루프가 Current State를 Desired로 계속 맞춥니다.
- 결과: YAML을 Git으로 관리하면 변경 이력/리뷰/롤백/재현성이 좋아집니다.

## Q6. 장애 복구 시나리오: 수동(Docker) vs 자동(Kubernetes)
- 운영 중 컨테이너 다운 시 수동 대응 vs Control Loop 기반 자동 복구

### 답변
- Docker 단독 운영에서는 장애 감지/재기동/원인 파악을 사람이 하게 되는 경우가 많습니다.
- Kubernetes에서는 Deployment/ReplicaSet 컨트롤러가 Desired(예: replicas=3)와 Current를 비교해 **자동으로 부족분을 복구(Self-healing)** 합니다.

## Q7. 개발자 관점의 3대 핵심 리소스 (Pod, Deployment, Service)

### 답변
- **Pod**: 컨테이너(들)의 실행 단위(스케줄링 최소 단위)
- **Deployment**: Pod 템플릿/개수 유지/업데이트(롤링)/롤백 등 “배포 관리”
- **Service**: Pod 집합에 대한 안정적인 접근점(고정 이름/가상 IP) + 로드밸런싱

## Q8. API Server를 중심으로 한 이벤트 기반 통신 (직접 제어 vs 상태 구독)

### 답변
- Kubernetes는 컴포넌트끼리 직접 명령하기보다 **API Server 상태를 Watch**하고, 상태 변경을 통해 협업합니다.
- 역할 분리
  - **Scheduler**: “배치만”(어느 노드에 둘지 결정)
  - **Controller Manager**: “개수/상태 유지”(Desired vs Current를 맞춤)
  - **Kubelet**: 실제 컨테이너 실행/상태 보고

## Q9. Control Plane의 고가용성(HA)과 SPoF 해결 (단일 서버인가, 다중 서버인가?)

### 답변
- 운영 환경은 보통 Control Plane을 다중화(3대+)하고 앞단에 LB를 둬 API Server 요청을 분산합니다.
- etcd도 쿼럼 기반(홀수 구성)으로 운영해 단일 장애점 위험을 낮춥니다.

## Q10. 내 터미널과 클러스터의 연결 고리, kubectl의 동작 경로

### 답변
- kubectl은 **API Server로 REST 호출하는 클라이언트**입니다.
- kubeconfig에 API Server 주소(운영은 보통 LB 주소)와 인증 정보가 들어 있어, kubectl이 그 정보를 읽고 요청을 보냅니다.

## Q11. Docker Swarm과 Kubernetes에서는 클러스터 단위의 이중화 기법있나?

### 답변
- **있습니다.**
- 큰 그림
  - **Swarm**: Manager 노드 다중화 + Raft 합의(쿼럼)로 클러스터 상태 유지
  - **Kubernetes**: Control Plane 다중화 + etcd 쿼럼 기반으로 상태 유지
