# 04/01 질문 답변

## 1) 외부 LB가 있을 때 LoadBalacner Type과 NodePort Type의 유의미한 차이가 무엇인지
- **핵심 차이**는 “Kubernetes가 외부 Load Balancer를 **자동으로 프로비저닝/연동**해주느냐”입니다.
- **Service `type: LoadBalancer`**
  - (클라우드 통합 환경에서) Service 생성 시 외부 LB를 자동 생성/연동합니다.
  - 서비스 단위로 외부 접근 엔드포인트(EXTERNAL-IP/DNS)가 생깁니다.
- **Service `type: NodePort`**
  - 외부 LB를 자동으로 만들지 않습니다.
  - 모든 노드에 동일 포트를 열어(`NodeIP:NodePort`) 외부 LB가 있다면 그 LB가 각 노드의 NodePort로 분산하도록 “별도로” 구성해야 합니다.

## 2) Service 타입이 NodePort일때 Node에 pod이 안떠있는경우에도 NodePort VM이 만들어지는지?
- NodePort는 **VM(노드)을 새로 만들지 않습니다.**
- NodePort Service를 생성하면 **클러스터의 모든 노드에서 해당 포트가 열리도록(프록시/룰 설정)** 동작합니다.
- 해당 노드에 Pod이 없어도(기본 `externalTrafficPolicy: Cluster` 기준) 요청이 **다른 노드에 있는 Pod으로 포워딩**될 수 있습니다.
- 단, Service 뒤에 엔드포인트(Pod)가 하나도 없으면 요청은 실패합니다.

## 3) K8s 서비스 타입 보안 비교: NodePort vs LoadBalancer, 프로덕션 환경에서는 NodePort 직접 노출을 피하고 LoadBalancer 또는 Ingress Controller를 사용하는 것이 권장되는 이유
- NodePort는 “모든 노드에 포트를 여는” 특성상
  - 노출 면(attack surface)이 넓어지고
  - 방화벽/보안그룹/포트 관리가 복잡해지고
  - 노드 IP/포트 기반 운영이 늘어나 “표준 진입점 통합”이 어려워질 수 있습니다.
- 그래서 프로덕션에선 보통
  - `LoadBalancer`로 서비스 단위 엔드포인트를 만들거나
  - **Ingress Controller로 진입점을 통합(대개 80/443)** 하는 패턴이 권장됩니다.
