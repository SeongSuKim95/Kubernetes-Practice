# 04/01 질문 답변

## 1) RollingUpdate가 아닌 Recreate로 하게되면 node의 리소스 부족으로 인한 prending이 안 일어나는지 궁금합니다.
- **아니요. Recreate에서도 Pending은 생길 수 있습니다.**
- 다만 Recreate는 “구버전 Pod과 신버전 Pod이 잠깐 공존하면서 추가 리소스가 필요한 구간(서지, surge)”이 없어서, RollingUpdate에 비해 **‘추가 리소스 때문에’ Pending이 생길 가능성은 줄어들 수** 있습니다.
- Recreate에서도 Pending이 생기는 대표 원인
  - 새 버전 Pod의 `resources.requests`가 더 커짐
  - 노드 자원이 여전히 부족(다른 워크로드가 점유)
  - `nodeSelector`/affinity/taints 같은 스케줄링 제약
  - PV 바인딩/쿼터/이미지 pull 등 다른 원인
- **주의**: Recreate는 보통 **다운타임이 발생**할 수 있어 “무중단 배포”와 충돌할 수 있습니다.

## 2) 무중단으로 kubenetes를 운영하려면 node에 여유분이 있어야할 거 같은데요. 실무에서는 보통 몇 퍼센트 정도 여유를 두고 운영하는지도 궁금하네요
- **고정된 정답 퍼센트는 없습니다.**
- 실무에선 “장애/드레인/업데이트” 같은 사건을 흡수하도록 **N+1(또는 AZ 단위 N+1)** 시나리오로 역산해 headroom을 잡는 경우가 많습니다.
- 일반 경향
  - **메모리**는 OOM 리스크가 커서 CPU보다 **더 보수적으로** 여유를 잡는 편이 많습니다.
- 결론: “몇 %”를 외우기보다, 목표 SLO와 장애 시나리오로 headroom을 설계하는 게 안전합니다.

## 3) 특정 Node에 Pod이 뜨도록 지정할 수 있는지?
가능합니다. 대표 옵션은 아래 3가지입니다.

- **(1) `nodeSelector`**
  - 노드 라벨과 정확히 매칭되는 노드에만 배치(가장 단순).
- **(2) `nodeAffinity` / `podAffinity` / `podAntiAffinity`**
  - “선호(preferred)” 또는 “필수(required)” 조건으로 더 정교하게 배치.
- **(3) `taints` / `tolerations`**
  - 특정 노드에 “기본적으로는 못 올라오게” 막고, toleration이 있는 Pod만 허용(격리/전용 노드 운영에 자주 사용).

주의: 이런 제약을 강하게 걸수록, 클러스터 여유가 부족할 때 **Pending 가능성**이 커집니다.
